## Becoming a Maintainer

We're always looking for dedicated contributors to join the maintainer team!

### Path to Maintainership

1. **Contribute Regularly**
   - Submit quality pull requests
   - Review others' contributions
   - Help with issues and discussions
   - Demonstrate expertise in an area

2. **Show Commitment**
   - Consistent contributions over 6+ months
   - Reliable and responsive
   - Align with project values
   - Positive community interactions

3. **Nomination**
   - Current maintainers nominate candidates
   - Candidate accepts nomination
   - Discussion period (2 weeks)
   - Vote by core maintainers (majority approval)

### Maintainer Expectations

#### Time Commitment
- **Core Maintainers**: 5-10 hours/week
- **Component Maintainers**: 3-5 hours/week
- **Flexibility**: We understand life happens; communicate availability

#### Responsibilities
- Review pull requests in your area
- Respond to issues and discussions
- Participate in maintainer meetings
- Follow the Code of Conduct
- Mentor new contributors

#### Privileges
- Write access to repository
- Ability to merge pull requests
- Influence on project direction
- Recognition in project materials
- Invitation to maintainer meetings

### Stepping Down

Maintainers can step down at any time:

1. Notify other maintainers
2. Transfer responsibilities
3. Update documentation
4. Optional: Move to emeritus status

No hard feelings - we appreciate all contributions!

## Decision Making

### Consensus Model

We strive for consensus on all decisions:

1. **Proposal**: Anyone can propose changes
2. **Discussion**: Open discussion period (minimum 1 week)
3. **Consensus**: Aim for agreement among maintainers
4. **Vote**: If consensus fails, majority vote by core maintainers
5. **Implementation**: Approved changes are implemented

### Types of Decisions

#### Minor Decisions (No Vote Required)
- Bug fixes
- Documentation improvements
- Code refactoring (no API changes)
- Test additions

#### Major Decisions (Consensus Required)
- Breaking API changes
- New major features
- Architecture changes
- Governance changes

#### Critical Decisions (Unanimous Vote)
- License changes
- Project name changes
- Removal of core features
- Maintainer removal (for cause)

### Conflict Resolution

If maintainers disagree:

1. **Discussion**: Extended discussion period
2. **Mediation**: Lead maintainer mediates
3. **Community Input**: Seek community feedback
4. **Vote**: Final decision by vote if needed
5. **Documentation**: Record decision and rationale

## Maintainer Guidelines

### Code Review

**Standards**:
- Review within 1 week (sooner for small PRs)
- Provide constructive feedback
- Test changes locally when possible
- Check for documentation updates
- Ensure tests pass

**Approval**:
- 1 approval required for minor changes
- 2 approvals required for major changes
- Core maintainer approval for breaking changes

### Issue Triage

**Response Time**:
- Critical bugs: 24 hours
- Regular issues: 1 week
- Feature requests: 2 weeks

**Labels**:
- Apply appropriate labels
- Set milestones for planned work
- Assign to maintainers when appropriate
- Close duplicates and off-topic issues

### Communication

**Channels**:
- GitHub Issues: Bug reports, feature requests
- GitHub Discussions: Questions, ideas, community
- Email: Private or sensitive matters
- Meetings: Complex discussions, planning

**Tone**:
- Professional and respectful
- Welcoming to newcomers
- Patient with questions
- Constructive with feedback

### Security

**Responsibilities**:
- Monitor security advisories
- Review security reports promptly
- Coordinate security releases
- Maintain confidentiality

**Contact**: security@langquality.org
